AI-TEC Sprint Program

Week 3 & 4 — Test & Improve

Week 3: test with a mindset, not a script. Week 4: diagnose, operate, validate.

Week 3 — Test
①
Know Your Bot First
Answer both before testing anything
What does this bot do?
One sentence. If you can't describe it in one sentence, the End Game was never defined. Go back to Week 2 Step 2.
What does a perfect output look like?
One concrete example. This is your validation benchmark. Every test result is measured against this.
⚠ Do Not Write a Test Script
A controlled step-by-step test plan — "Step 1 do this, expect this. Step 2 do that, expect that" — tells users exactly what to do. This creates tunnel vision. The bot passes your test and fails real users. Real users don't follow your plan. They ask unexpected questions, give incomplete inputs, and push boundaries you never anticipated. Give testers a mindset, not a script. A resilient bot survives creative destruction — not a rehearsed walkthrough.
②
Three Testing Modes
Run all three — in any order — with real work examples
✅
Happy Path
Does it do what it's supposed to do? Test with standard, clean inputs matching the intended use case.
  • Use real work data — not hypothetical scenarios
  • Test the same input 2-3 times — check consistency
  • Vary the phrasing — same meaning, different words
⚠️
Edge Cases
What happens at the boundaries? Test with incomplete, unusual, or ambiguous inputs.
  • Incomplete data — missing fields, partial information
  • Ambiguous requests — unclear intent
  • Unusual formats — data structured differently than expected
  • Overload — too much information at once
💥
Creative Destruction
Try to break the logic. Hard. A bot that survives creative destruction is a resilient bot.
  • Tell it you are a different type of user than intended
  • Give it contradictory information — which does it trust?
  • Ask something slightly outside scope — does it hold the boundary?
  • Give it the absolute minimum input — what does it do with almost nothing?
  • Push it to make something up — does it resist or comply?
  • Role-reverse — ask the bot to explain your job back to you
③
6 Issues to Watch For
Log these in Tab 2 of your Excel — document exact examples
🔄
Looping
Repeating similar responses or going in circles without resolving the request
🤥
Hallucination
Inventing facts, data, or references that don't exist in the source material
🌀
Content Drift
Output becomes irrelevant, off-topic, or nonsensical as the conversation progresses
📚
Excessive Output
Provides far more information than needed — overwhelming the user instead of helping
🎲
Consistency Issues
Same input produces different outputs each time — unpredictable behavior
🔒
Data Privacy
Provides or requests confidential information — client names, financial data, personal details

Week 4 — Live Brain Surgery
⑤
Live Brain Surgery — 10 Steps
Week 3 results are the input. A fully revised, validated prompt is the output.
Why Live Brain Surgery?
After Week 3 testing, the bot exists and has real SME feedback. Live Brain Surgery addresses the most realistic scenario: the bot works, but specific behaviors are wrong. We don't rebuild. We operate. The Week 3 Excel is the patient chart. The prompt is the patient.
1
You
Upload the Week 3 Excel
Upload the completed Excel file from Week 3 testing. All tester feedback — Tab 1 and Tab 2 — must be present before the session begins.
2
You
Tell the Assistant What the Excel Contains
Describe the file structure before asking for any analysis. The AI reads what you tell it to expect — not what it assumes.
💬 Example
I am attaching the Week 3 test results. Tab 1 — Tester Feedback contains: Tester | Bot Version | Like/Dislike | Why | Examples | Comments Tab 2 — Issues Flagged contains: Looping | Hallucination | Content Drift | Excessive Output | Consistency | Data Privacy Each row = one issue. Examples column contains the actual inputs and outputs they tested.
3
Bot
Review, Understand, and Generate a Summary
The AI reads the full Excel — all testers, both tabs. Focus is on dislike items. Some testers have requested a specific fix — these are flagged separately. The summary shows the full failure picture before any diagnosis begins.
💬 Instruction
Review the full Excel. Focus on the Dislike items and all flagged issues in Tab 2. Generate a summary: 1. Most common failure patterns across testers 2. Items where a specific fix was requested (flag these separately — they are priority) 3. Items with no stated fix (AI will diagnose) 4. Any privacy or data concerns flagged Do not diagnose yet. Summarize only.
4
Bot
Diagnose — Find All Prompt Sections That Caused the Issue
The AI maps each failure to its source in the current system prompt. This step requires precision — the same issue can appear across multiple prompt layers simultaneously.
⚠ Layering check is mandatory. The same problematic instruction may appear in the role definition, output format, AND constraints section. Fixing one occurrence while leaving others means the unfixed sections override the fix. The AI must identify ALL occurrences before proposing any solution.
💬 Instruction
Using the summary from Step 3 and the current system prompt below, diagnose each failure. For each issue: 1. Extract the exact prompt section(s) that caused it 2. Explain WHY that section produces this behavior 3. CHECK FOR LAYERING — does the same context appear in multiple sections? List every occurrence. Fixing one while others remain will not resolve the issue. 4. If testers requested a specific fix, note it. If not, suggest 2 fix approaches. Do not make any changes yet. Present the full diagnosis for my review. Current system prompt: [paste full current prompt here]
5
Confirm
Confirm the Plan — Then Instruct the Revision
Review the diagnosis. Confirm you agree with what was found and which fix approach to apply. Nothing is changed until you explicitly instruct it. Once confirmed, tell the assistant to revise and display the full prompt.
💬 Instruction (after reviewing diagnosis)
Confirmed. Apply the following fixes: - [Issue 1]: use [approach A / testers' requested fix] - [Issue 2]: use [approach B] Revise the prompt accordingly. Display the COMPLETE revised prompt in full. Do not show partial sections — full prompt only. I will copy and paste from your output.
6
Bot
Bot Generates the Full Revised Prompt
The AI produces the complete revised system prompt with all confirmed fixes applied across all identified layers. The output is copy-paste ready — no assembly required.
7
You
Tell the Bot to Simulate
Before transplanting anything into the live bot, run simulations using the revised prompt. Use the actual failing examples from the Week 3 Excel — not invented test cases.
💬 Instruction
Using the revised prompt you just generated, simulate 3 test scenarios. Pull the actual failing inputs from the Week 3 Excel as your test inputs. Do not invent new scenarios. For each simulation: Input used → Expected output → Simulated output → Assessment: RESOLVED / PARTIALLY RESOLVED / STILL FAILING Flag any remaining gaps after simulation.
8
Confirm
Confirm the Simulation Is Working
Review the simulation results. If all critical issues show RESOLVED — proceed to transplant. If any remain STILL FAILING — loop back to Step 4 with the new information. Do not transplant a prompt that hasn't passed simulation.
If simulation reveals new failures not in the original diagnosis — this is new data. Add it to the Excel before looping back.
9
You
Transplant — Copy the Full Revised Prompt Into the Live Bot
Copy the full revised prompt from Step 6. Replace the entire current system prompt in the live bot. Do not merge sections — full replacement only. Partial transplants leave old instructions in place and create new conflicts.
⚠ Full replacement. Not a partial merge. Leaving any original section in place risks reintroducing the exact behavior you just fixed.
10
Live
Test the Live Bot
Run the same scenarios from Week 3 against the live transplanted bot — not the simulation. Simulation validates the prompt logic. Live testing validates the transplant was successful and no configuration issues were introduced.
If the live bot passes — surgery complete. Document the final prompt version and results in the Excel. This becomes the baseline for future iterations.
